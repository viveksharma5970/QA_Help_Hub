TestNG Framework

TestNG – Testing Framework for Java

TestNG is a powerful testing framework for Java inspired by JUnit and NUnit. It provides flexible test configuration, parallel execution, data-driven testing, and rich reporting – making it ideal for enterprise automation suites.

  • Group tests, dependencies, priorities
  • Data-driven & parameterized tests
  • Flexible lifecycle hooks & listeners

Complete TestNG Annotations Reference

Suite, test, class, method, group level hooks + configuration annotations.

Lifecycle Annotations

Suite / Test Level

BS

@BeforeSuite

Runs once before all tests in the entire suite.

Use for global setup such as reading config files, initializing reporting, or starting containers. 

@BeforeSuite
public void beforeSuite() {
    // Global setup before any tests run
}
AS

@AfterSuite

Runs once after all tests in the suite.

Ideal for closing global resources, generating final reports, or sending notifications. 

@AfterSuite
public void afterSuite() {
    // Global cleanup after all tests are done
}
BT

@BeforeTest

Runs before any test method in a <test> in testng.xml.

Good place for per-test-block setup such as environment or browser configuration. 

@BeforeTest
public void beforeTestBlock() {
    // Runs before <test> block in XML
}
AT

@AfterTest

Runs after all test methods in a <test> block.

Useful for test-level reporting or environment-specific teardown. 

@AfterTest
public void afterTestBlock() {
    // Runs after all methods in this <test> block
}

Class / Method / Group Level

BC

@BeforeClass

Runs once before the first test method in the class.

Typically used to initialize WebDriver or page objects at class level. 

@BeforeClass
public void setUpClass() {
    // Initialize resources shared by all tests in this class
}
AC

@AfterClass

Runs once after all test methods in the class.

Commonly used to quit WebDriver or free class-level resources. 

@AfterClass
public void tearDownClass() {
    // Cleanup after all tests in this class
}
BM

@BeforeMethod

Runs before each @Test method.

Often used to navigate to a page, reset data, or login before each test. 

@BeforeMethod
public void beforeMethod() {
    // Runs before each @Test method
}
AM

@AfterMethod

Runs after each @Test method.

Good for capturing screenshots on failure, logging, or resetting state. 

@AfterMethod
public void afterMethod() {
    // Runs after each @Test method
}
BG

@BeforeGroups

Runs before the first test method of specified groups.

Useful for preparing context only when certain groups are executed (e.g. smoke, regression). 

@BeforeGroups("smoke")
public void beforeSmokeGroup() {
    // Runs before first test in 'smoke' group
}
AG

@AfterGroups

Runs after all test methods in specified groups.

Great for group-specific cleanup, logging, or report aggregation. 

@AfterGroups("smoke")
public void afterSmokeGroup() {
    // Runs after all tests in 'smoke' group
}

Test Configuration Annotations

T

@Test

Marks a method as a test with rich configuration.

Control execution using priority, groups, dependsOnMethods, invocationCount, timeOut, enabled and alwaysRun. 

@Test(
    priority = 1,
    groups = {"smoke"},
    enabled = true,
    invocationCount = 2,
    timeOut = 5000,
    dependsOnMethods = "loginTest",
    alwaysRun = false
)
public void dashboardTest() {
    // Test implementation
}
DP

@DataProvider

Supplies multiple sets of data to a test method.

Ideal for testing the same logic with many inputs – returns Object[][] or Iterator<Object[]>. 

@DataProvider(name = "loginData")
public Object[][] getLoginData() {
    return new Object[][] {
        {"user1", "pwd1"},
        {"user2", "pwd2"}
    };
}

@Test(dataProvider = "loginData")
public void loginTest(String username, String password) {
    // Use provided credentials
}
P

@Parameters

Injects values from testng.xml into methods.

Useful for environment URLs, browser types, or credentials without hardcoding. 

@Parameters({"url", "browser"})
@Test
public void launchApp(String url, String browser) {
    // Use values from testng.xml
}
F

@Factory

Creates test class instances dynamically.

Helpful for running same tests against multiple data sets or configurations at class level. 

@Factory
public Object[] factoryMethod() {
    return new Object[] {
        new LoginTest("chrome"),
        new LoginTest("firefox")
    };
}
L

@Listeners

Attach custom listeners for reporting and hooks.

Implement ITestListener, ISuiteListener etc. for logs, screenshots, or dashboards. 

@Listeners(ExtentReportListener.class)
public class LoginTests {
    // Listener hooks for reporting
}
O

@Optional

Provides default value for @Parameters.

Prevents failures when parameter is missing in testng.xml. 

@Parameters("env")
@Test
public void testEnv(@Optional("qa") String env) {
    // Will use "qa" if env is not passed
}
AR

alwaysRun

Force configuration methods to always execute.

Use for cleanup or logging that must run even when dependent tests are skipped or fail. 

@AfterMethod(alwaysRun = true)
public void tearDown(ITestResult result) {
    // Always executed for logging / screenshots
}

TestNG Assertions & Utility Methods

Hard / soft assertions, Reporter API, and execution control parameters.

Click to expand
As

Assert.assertEquals / assertTrue / assertFalse

Standard hard assertions that fail the test immediately.

Use for critical validations where failing fast is desired. 

@Test
public void loginTitleTest() {
    String actual = driver.getTitle();
    Assert.assertEquals(actual, "Dashboard", "Page title mismatch");
    Assert.assertTrue(actual.contains("Dash"));
    Assert.assertFalse(actual.contains("Error"));
}

assertNull / assertNotNull / assertThrows

Validate nullability and expected exceptions in your tests. 

@Test(expectedExceptions = IllegalArgumentException.class)
public void exceptionTest() {
    Assert.assertNull(getUserIfNotExists());
    Assert.assertNotNull(getExistingUser());
    throw new IllegalArgumentException("Invalid argument");
}

SoftAssert

Collect multiple assertion failures and report them together using assertAll(). 

@Test
public void profileValidation() {
    SoftAssert soft = new SoftAssert();
    soft.assertEquals(actualName, "Vivek", "Name mismatch");
    soft.assertTrue(isEmailValid(email), "Email format invalid");
    soft.assertAll(); // Important to trigger failures
}

Reporter.log()

Log custom messages to TestNG reports – very handy for debugging and reporting. 

@Test
public void sampleTest() {
    Reporter.log("Starting sampleTest", true);
    // Test steps...
    Reporter.log("sampleTest completed", true);
}

dependsOnMethods / priority / enabled / invocationCount / timeOut

Fine-grained control on how and when tests execute. 

@Test(priority = 0)
public void login() { }

@Test(priority = 1, dependsOnMethods = "login", timeOut = 3000)
public void openDashboard() { }

@Test(enabled = false)
public void deprecatedTest() { }

@Test(invocationCount = 3)
public void retryableAction() { }
Selenium WebDriver

Selenium WebDriver – Browser Automation Engine

Selenium WebDriver is the de-facto standard for web UI automation. It follows a client–server architecture where your test code (client) communicates with the browser driver (server) to simulate real user interactions.

  • Supports multiple browsers & programming languages
  • Works with grid/cloud for parallel execution
  • Rich API for elements, waits, alerts, windows, frames, and actions

WebDriver Browser Commands

Navigation, window management, and browser control methods

get() / navigate().to()

Open URL in browser

driver.get("https://example.com");
driver.navigate().to("https://example.com");

navigate().back() / forward() / refresh()

Browser history navigation

driver.navigate().back();
driver.navigate().forward();
driver.navigate().refresh();

getTitle() / getCurrentUrl() / getPageSource()

Get page information

String title = driver.getTitle();
String url = driver.getCurrentUrl();
String source = driver.getPageSource();

close() / quit()

Close window or quit session

driver.close(); // closes current window
driver.quit();  // quits entire session

Synchronization Methods & Waits

Implicit, Explicit, and Fluent waits for handling dynamic elements

Implicit Wait

Global wait applied to all findElement calls

driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));

Explicit Wait - WebDriverWait

Wait for specific condition on specific element

WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
WebElement element = wait.until(ExpectedConditions.visibilityOfElementLocated(By.id("username")));

Fluent Wait

Wait with polling interval and ignore exceptions

Wait wait = new FluentWait<>(driver)
    .withTimeout(Duration.ofSeconds(30))
    .pollingEvery(Duration.ofSeconds(2))
    .ignoring(NoSuchElementException.class);
WebElement element = wait.until(driver -> driver.findElement(By.id("dynamic")));

ExpectedConditions

Common wait conditions

ExpectedConditions.visibilityOfElementLocated(By.id("id"));
ExpectedConditions.elementToBeClickable(By.id("btn"));
ExpectedConditions.presenceOfElementLocated(By.xpath("//div"));
ExpectedConditions.invisibilityOfElementLocated(By.id("loader"));
ExpectedConditions.titleContains("Dashboard");

Window & Frame Handling

Window Handling

String mainWindow = driver.getWindowHandle();
Set allWindows = driver.getWindowHandles();
for(String window : allWindows) {
    driver.switchTo().window(window);
}
driver.switchTo().window(mainWindow);

Frame Handling

driver.switchTo().frame(0); // by index
driver.switchTo().frame("frameName"); // by name
driver.switchTo().frame(element); // by WebElement
driver.switchTo().defaultContent(); // back to main

Screenshots & JavascriptExecutor

Take Screenshot

TakesScreenshot ts = (TakesScreenshot) driver;
File source = ts.getScreenshotAs(OutputType.FILE);
FileUtils.copyFile(source, new File("./screenshot.png"));

JavascriptExecutor

JavascriptExecutor js = (JavascriptExecutor) driver;
js.executeScript("alert('Hello');");
js.executeScript("arguments[0].click();", element);
js.executeScript("arguments[0].value='text';", element);

Scroll Operations

Scroll Page

JavascriptExecutor js = (JavascriptExecutor) driver;
// Scroll down
js.executeScript("window.scrollBy(0,500)");
// Scroll to bottom
js.executeScript("window.scrollTo(0, document.body.scrollHeight)");
// Scroll to element
js.executeScript("arguments[0].scrollIntoView(true);", element);
Cucumber & Gherkin

BDD with Cucumber – Living Documentation for Automation

Cucumber enables Behaviour-Driven Development (BDD) using human-readable .feature files written in Gherkin. Devs, QA and business can collaborate on the same scenarios, while Java step definitions automate them via Selenium or API clients.

  • Readable scenarios: Given / When / Then
  • Powerful hooks & tags to organize suites
  • Rich runner options & plugins for reports

Gherkin Syntax & Feature File Structure

Core building blocks of Cucumber scenarios that map to Java step definitions.

Feature / Background / Scenario / Scenario Outline / Examples

A Feature groups related scenarios. Use Background for common pre-steps, Scenario for single flows, and Scenario Outline with Examples for data-driven variations. 

Feature: Login
  Background:
    Given user is on login page

  Scenario: Valid login
    When user logs in with "vivek" and "pwd123"
    Then dashboard should be visible

  Scenario Outline: Invalid login
    When user logs in with "<user>" and "<pass>"
    Then error message "<message>" should be shown

    Examples:
      | user  | pass   | message              |
      | foo   | wrong  | Invalid credentials  |
      | admin | empty  | Password is required |

Given / When / Then / And / But

Gherkin step keywords that describe preconditions (Given), actions (When) and outcomes (Then). And and But continue the previous type. 

Scenario: Transfer money
  Given account "123" has balance 1000
  And account "456" has balance 500
  When user transfers 200 from "123" to "456"
  Then account "123" should have balance 800
  And account "456" should have balance 700

@Tags / DataTable / DocString

Use @smoke, @regression tags to group scenarios. Tables become DataTable in steps, and triple-quoted blocks act as multi-line DocString. 

@smoke @ui
Scenario: Bulk user creation
  Given the following users exist:
    | name  | role  |
    | Vivek | admin |
    | John  | user  |
  """
  Additional notes can go
  here as a DocString.
  """

Step Definitions, Hooks & Runner Options

Java glue code that binds Gherkin steps to automation plus lifecycle hooks and runner config.

@Given / @When / @Then / @And

Map Gherkin steps to Java methods using either regular expressions or Cucumber Expressions. Method parameters capture data from the step. 

public class LoginSteps {

  @Given("user is on login page")
  public void userOnLoginPage() {
      // open login page
  }

  @When("user logs in with {string} and {string}")
  public void userLogsIn(String username, String password) {
      // perform login
  }

  @Then("dashboard should be visible")
  public void dashboardVisible() {
      // assertion
  }
}

DataTable & Parameter Types

Use io.cucumber.datatable.DataTable to consume tables and define custom @ParameterType to map text to domain objects. 

@When("the following users exist:")
public void usersExist(DataTable table) {
    List<Map<String, String>> rows = table.asMaps();
    // iterate rows and create users
}

@Before / @After / @BeforeStep / @AfterStep

Hooks run before/after scenarios or steps. Use order and tags to control execution and scope (e.g., only UI scenarios initialize WebDriver). 

public class Hooks {

  @Before(order = 0)
  public void globalSetup() {
      // runs before every scenario
  }

  @Before("@ui")
  public void uiSetup() {
      // e.g. start WebDriver
  }

  @After(order = 1)
  public void tearDown(io.cucumber.java.Scenario scenario) {
      // take screenshot on failure, close driver
  }

  @BeforeStep
  public void beforeEachStep() {
      // fine-grained logging
  }
}

@CucumberOptions & JUnit/TestNG Runner

Configure where feature files live, glue packages, reporting plugins, tags to include/exclude, and dry run behaviour. 

@RunWith(io.cucumber.junit.Cucumber.class)
@io.cucumber.junit.CucumberOptions(
    features = "classpath:features",
    glue = {"steps", "hooks"},
    plugin = {"pretty", "html:reports/cucumber.html"},
    tags = "@smoke and not @wip",
    monochrome = true,
    dryRun = false,
    publish = true
)
public class CucumberTestRunner {
}

maven-surefire-plugin & maven-failsafe-plugin

surefire runs unit / TestNG tests in test phase; failsafe is ideal for integration / Cucumber suites in integration-test phase.

    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-surefire-plugin</artifactId>
      <version>3.2.5</version>
      <configuration>
        <suiteXmlFiles>
          <suiteXmlFile>testng.xml</suiteXmlFile>
        </suiteXmlFiles>
      </configuration>
    </plugin>

    <!-- Optional: separate Cucumber integration tests -->
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-failsafe-plugin</artifactId>
      <version>3.2.5</version>
      <executions>
        <execution>
          <goals>
            <goal>integration-test</goal>
            <goal>verify</goal>
          </goals>
          <configuration>
            <includes>
              <include>**/Cucumber*Runner.java</include>
            </includes>
          </configuration>
        </execution>
      </executions>
    </plugin>
  </plugins>
</build>

Apache POI - Excel File Handling

XSSF (xlsx) and HSSF (xls) APIs for reading and writing Excel files

XSSF API - Excel 2007+ (.xlsx)

Read Excel File (XSSF)

XSSFWorkbook, XSSFSheet, XSSFRow, XSSFCell

FileInputStream fis = new FileInputStream("data.xlsx");
XSSFWorkbook workbook = new XSSFWorkbook(fis);
XSSFSheet sheet = workbook.getSheetAt(0);
XSSFRow row = sheet.getRow(0);
XSSFCell cell = row.getCell(0);
String value = cell.getStringCellValue();
workbook.close();

Write Excel File (XSSF)

Create workbook, sheet, row, and cell

XSSFWorkbook workbook = new XSSFWorkbook();
XSSFSheet sheet = workbook.createSheet("TestData");
XSSFRow row = sheet.createRow(0);
XSSFCell cell = row.createCell(0);
cell.setCellValue("Test Value");
FileOutputStream fos = new FileOutputStream("output.xlsx");
workbook.write(fos);
workbook.close();

Important XSSF Methods

// Workbook methods
workbook.getNumberOfSheets();
workbook.getSheet("SheetName");
workbook.createSheet("NewSheet");

// Sheet methods
sheet.getLastRowNum();
sheet.getRow(rowIndex);
sheet.createRow(rowIndex);

// Row methods
row.getLastCellNum();
row.getCell(cellIndex);
row.createCell(cellIndex);

// Cell methods
cell.getStringCellValue();
cell.getNumericCellValue();
cell.getBooleanCellValue();
cell.setCellValue("value");

Important HSSF Methods

// Same methods as XSSF
// Workbook, Sheet, Row, Cell methods are identical
// Only difference is class names: HSSF vs XSSF

// Cell Type Check

}
if(cell.getCellType() == CellType.NUMERIC) {
    double val = cell.getNumericCellValue();
}
if(cell.getCellType() == CellType.BOOLEAN) {
    boolean val = cell.getBooleanCellValue();
}

REST Assured

REST Assured – Fluent API Testing for REST Services

REST Assured provides a fluent DSL for validating HTTP APIs. Combine it with TestNG and Cucumber to build reliable, readable API automation suites.

  • Fluent given / when / then style
  • Supports JSON, XML, auth, params, and filters
  • Seamless integration with Hamcrest & JSONPath

Core REST Assured API – given() / when() / then()

Main fluent entry point using io.restassured.RestAssured and related interfaces.

RestAssured.given() / when() / then()

Typical static imports: import static io.restassured.RestAssured.*; and import static org.hamcrest.Matchers.*;. Chain methods to build request, execute, and assert. 

import static io.restassured.RestAssured.*;
import static org.hamcrest.Matchers.*;

@Test
public void getUserDetails() {
    given()
        .baseUri("https://api.example.com")
        .basePath("/users/{id}")
        .pathParam("id", 1)
    .when()
        .get()
    .then()
        .statusCode(200)
        .body("name", equalTo("Vivek"))
        .body("roles", hasItem("ADMIN"));
}

Request & Response Objects

Configure RequestSpecification, inspect Response, and chain ValidatableResponse.

RequestSpecification – headers, params, body, auth

Use given() or a reusable RequestSpecification to configure base URI, authentication and common headers. 

RequestSpecification req = given()
    .baseUri("https://api.example.com")
    .basePath("/users")
    .contentType("application/json")
    .auth().basic("user", "pwd")
    .header("x-api-key", "secret")
    .queryParam("active", true);

Response response = req
    .body("{\"name\":\"Vivek\"}")
    .post();

Response & ValidatableResponse

Use Response to log and inspect raw payload; use then() to obtain a ValidatableResponse for assertions. 

Response res = given()
    .baseUri("https://api.example.com")
    .when()
    .get("/health");

res.then()
    .statusCode(200)
    .log().body();

given()

.when()
    .get("/users")
.then()

public class ApiBase {

Postman – API Client

Postman – Designing, Testing & Documenting REST APIs

Postman is a powerful GUI client for creating, running and automating REST API requests. Use it to explore endpoints, validate responses, and generate collections that can be run in CI.

  • Supports all HTTP methods, auth types, headers & cookies
  • Built-in test runner using the pm API
  • Environment variables, scripts and collections for automation

REST Methods – Building Requests in Postman

Step-by-step usage for GET, POST, PUT, DELETE and PATCH with params, body and headers.

GET – Read data

Select GET, enter URL (with path params), and use the Params tab for query parameters like ?page=1&size=20. 

// Example
GET https://api.example.com/users?page=1&size=20

// Steps in Postman:
// 1. Method: GET
// 2. URL: https://api.example.com/users
// 3. Params tab: page=1, size=20
// 4. Headers tab: Accept=application/json
// 5. Click "Send" and inspect Status, Time, Size and JSON body.

POST – Create data

Select POST, then in Body choose raw & JSON to send a JSON payload. 

// Example
POST https://api.example.com/users
Content-Type: application/json

{
  "name": "Vivek",
  "email": "vivek@example.com"
}

// Steps in Postman:
// 1. Method: POST, URL: https://api.example.com/users
// 2. Headers: Content-Type = application/json
// 3. Body tab: raw + JSON, paste JSON body
// 4. Send and verify 201 Created and response body.

PUT / PATCH – Update data

Use PUT to replace resources and PATCH for partial updates. Configure URL with ID, headers and JSON body. 

// Example
PUT https://api.example.com/users/1
Body:
{
  "name": "Vivek Updated"
}

DELETE – Remove data

Select DELETE, specify resource URL and send. Usually expect 204 No Content or 200 OK. 

// Example
DELETE https://api.example.com/users/1
// Steps: Method DELETE, URL with ID, click Send and check status.

Tests Tab – Using the pm Object

Write JavaScript assertions inside the Tests tab using the Postman pm API.

Basic tests – status, headers, body

Use pm.test(name, function) with pm.expect or built-in checks to validate responses. 

// Tests tab example
pm.test("Status code is 200", function () {
    pm.response.to.have.status(200);
});

pm.test("Response time < 1s", function () {
    pm.expect(pm.response.responseTime).to.be.below(1000);
});

pm.test("Content-Type is JSON", function () {
    pm.response.to.have.header("Content-Type");
    pm.expect(pm.response.headers.get("Content-Type")).to.include("application/json");
});

Body assertions & extracting values

Use pm.response.json() to parse JSON and store fields in environment or global variables for later requests. 

// In Tests tab
const json = pm.response.json();

pm.test("User name present", function () {
    pm.expect(json.name).to.exist;
});

// Save id to environment
pm.environment.set("userId", json.id);

// Typical setups

Environments, Variables & Scripts

Parameterise collections with environments and use pre-request / test scripts to chain calls.

Environment & Global Variables

Create environments (e.g. dev, qa) and reference variables with {{variableName}} inside URLs, headers, and bodies. 

// In request URL
{{baseUrl}}/users/{{userId}}

// In Tests script, store values
pm.environment.set("userId", pm.response.json().id);
const current = pm.environment.get("userId");

Pre-request scripts

Run JavaScript before each request to set dynamic headers, tokens or timestamps. 

// Pre-request Script example
const uuid = pm.variables.replaceIn("{{$guid}}");
pm.environment.set("correlationId", uuid);

pm.request.headers.add({
  key: "x-correlation-id",
  value: uuid
});
